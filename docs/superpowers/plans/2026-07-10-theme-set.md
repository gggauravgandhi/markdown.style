# Theme Set Implementation Plan (Plan 3)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Grow the theme set from 1 (paper) to 8, seven new themes with distinct visual personalities, each satisfying the Plan 1 theme contract, plus a per-theme render smoke test that makes invalid Shiki/Mermaid theme names unshippable.

**Architecture:** Pure additive work: 7 new CSS files in `src/themes/` + registry entries. The existing hygiene tests (`registry.test.ts` uses `it.each(themes)`) automatically extend to every new theme. A new `themes-render.test.ts` renders a code-fenced document through `render()` for EVERY registry theme; this is the safety net for Shiki theme-name validity (an invalid name throws inside `highlightFences`; the fallback there retries the language, not the theme).

**Tech Stack:** CSS only + registry entries. No new dependencies. System font stacks ONLY (self-containment: exported HTML must never reference webfonts).

**Spec:** `docs/superpowers/specs/2026-07-10-markdown-style-design.md` (section 3)

## Global Constraints

- Use `bun` for everything, never npm/npx/yarn/node.
- **Theme contract (hygiene-tested):** every theme CSS defines `--mds-bg`, `--mds-fg`, `--mds-font-body`, `--mds-font-heading` and contains an `@media print` block; `defaultAccent` matches `^#[0-9a-fA-F]{6}$`.
- **System font stacks only**, no `@import`, no `url()`, no webfonts in any theme CSS (self-containment invariant; the render suite asserts no non-data `url()` in output).
- All tunables via `--mds-*` custom properties; knobs (`--mds-accent`, `--mds-font-scale`, `--mds-page-width`) must work identically across all themes (never hardcode accent colors where `var(--mds-accent)` belongs).
- Content stays plain flowing blocks, no flex/grid wrappers around content blocks (Paged.js constraint).
- Each theme styles the full markdown element set: h1–h6, p, links, lists (incl. task lists), tables, blockquote, inline code + `pre.shiki` blocks, hr, images, `.mds-mermaid`, `.mds-error`, `.footnotes`.
- `shikiTheme` must be a REAL bundled Shiki theme name; `mermaidTheme` must be one of the `MermaidTheme` union. The per-theme render test enforces the former at test time.
- Dark themes must remain printable: their `@media print` block flips to a white page (`--mds-bg: #ffffff`) with dark text. Shiki code blocks keep their own inline backgrounds; that's acceptable (print-color-adjust is set in base).
- `paper` stays FIRST in the registry (it's the app default). New themes append after it.
- TDD: each task first extends the failing expectations (theme ids + count), then adds the CSS/registry entries to pass.
- Conventional commits, no Co-Authored-By.

- [ ] **Step 1: Write the per-theme render test (passes for `paper` today; it pins the harness before new themes land)**

`src/themes/themes-render.test.ts`:
```ts
import { describe, expect, it } from 'vitest'
import { render } from '../pipeline/render'
import { themes } from './registry'

const SMOKE_DOC = `# Theme Smoke

Body text with a [link](https://example.com), \`inline code\`, and **bold**.

| Col | Val |
|-----|-----|
| a   | 1   |

- [x] task done

\`\`\`ts
const n: number = 1
\`\`\`

> quoted line
`

// Renders through the REAL pipeline per theme. This is the enforcement point
// for shikiTheme validity: an invalid bundled-theme name throws inside
// highlightFences (the fallback there retries the language, not the theme).
describe('every registry theme renders end-to-end', () => {
  it.each(themes.map(t => [t.id, t] as const))('theme %s', async (_id, theme) => {
    const { html, errors } = await render(SMOKE_DOC, theme.id)
    expect(errors).toEqual([])
    expect(html).toContain('shiki')                    // code fence highlighted
    expect(html).toContain(theme.css.slice(0, 40))     // this theme's css inlined
    expect(html).not.toMatch(/url\(\s*['"]?(?!data:)/) // self-contained (no webfonts)
    expect(html).not.toMatch(/<script/i)
  })
})
```

## Self-Review Notes

- The per-theme render smoke test (Task 1) is the plan's real safety net: it executes Shiki with each theme's `shikiTheme` name against the actual bundled-themes list, so a typo'd theme name fails tests instead of throwing at runtime in the app.
- Carbon's print block flips the page to white, the one theme where screen and print palettes intentionally diverge; Shiki blocks keep their baked dark background (accepted; base sets print-color-adjust).
- Scholar justifies body text; headings/lists/tables/footnotes explicitly reset to left to avoid ragged justification artifacts.
- All font stacks are system-only; the render smoke test's `url()` assertion enforces the no-webfont invariant mechanically.
- Pop's h1 uses a solid accent color (an earlier gradient + `background-clip: text` draft was dropped; it prints unpredictably and reads as transparent in some print engines).
- Contrast's yellow accent is used for rules/markers/underlines only, never as text on white (a11y).
- `defaultAccent` values are all 6-digit hex (hygiene-tested).
