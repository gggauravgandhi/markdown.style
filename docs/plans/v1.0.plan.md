You are working on **markdown.style**, an open-source, entirely client-side web application that turns AI-generated Markdown into polished, professional documents.

Your task is to improve the public-facing website—primarily the homepage—by implementing a clearer content hierarchy, stronger positioning, a direct Markdown entry experience, better product proof, and balanced SEO.

Do not blindly rewrite or redesign everything. First inspect the existing codebase, understand its framework, routing, design system, editor state management, file-opening logic, export functionality, themes registry, existing example document, metadata implementation, and reusable components. Then implement the changes cleanly within the existing architecture.

---

# 1. Product definition

markdown.style helps users take Markdown generated by ChatGPT, Claude, Gemini, Copilot, or another AI assistant and turn it into a polished document suited to its content and audience.

Typical documents include:

* Technical documentation
* Software architecture documents
* Product requirement documents
* Research reports
* Academic papers
* Business reports
* Proposals
* Decision documents
* Long-form professional content

The user should generate the content with AI once, then use markdown.style for presentation, editing, iteration, and exporting.

This product is not primarily:

* A full-featured Markdown writing application
* A replacement for Obsidian, Typora, Notion, or VS Code
* A generic file converter
* A theme showcase
* An AI writing tool
* A server-side document processor

It is best positioned as a **styling and export workspace for AI-generated Markdown**.

# 2. Core positioning

The hierarchy of the homepage message must be:

## Primary benefit

**Turn AI-generated Markdown into a polished, professional document.**

The first thing users should understand is that their useful but visually plain AI output can become a credible document that they can download and share.

## Secondary benefit

**Generate the content once. Do not keep spending AI tokens regenerating HTML or PDFs.**

Users commonly ask an LLM to:

* Convert its Markdown answer to HTML
* Design that HTML
* Change the design
* Fix the PDF layout
* Regenerate the document after a small content edit
* Recreate the entire response just to alter formatting

markdown.style eliminates that loop.

Users can:

* Generate the content once
* Paste or open the Markdown
* Edit it locally
* Change the document style
* Change the accent colour
* Adjust font size
* Preview changes instantly
* Export again without another AI request

Be technically accurate. Prefer wording such as:

* “No additional AI tokens”
* “Without asking the AI to regenerate it”
* “Style and export without another prompt”

Do not make misleading claims such as implying that using the application itself consumes AI tokens.

## Supporting benefits

* Purpose-built document styles for different audiences and document types
* Broad support for rich Markdown commonly produced by AI
* Direct PDF download
* HTML export
* Local-only processing
* No signup
* No usage limits
* Free
* Open source

Privacy is important reassurance, but it is not the leading message.

Themes are an important capability, but they are not the central product promise.

# 8. Homepage Markdown input gateway

The hero should contain a lightweight way to begin—not the full editor interface.

Users should be able to:

* Paste Markdown into a textarea
* Drag and drop a Markdown file
* Open a file using the system file picker
* Continue into the existing editor with the content loaded

The recommended interface is a large, polished input/drop zone with wording similar to:

> Paste your Markdown here…

Supporting actions:

* Drop a `.md` file
* Open a file
* Optionally: Try an example

After valid content is present, show a strong action:

> Style this Markdown

Do not automatically redirect immediately after paste. The user should remain in control and activate the button.

## Required behaviour

* Reuse the existing editor’s document-loading/state mechanism wherever possible.
* Do not create a parallel Markdown parsing or persistence implementation.
* Preserve the pasted/opened content when navigating to the editor.
* Use the cleanest existing mechanism: shared state, localStorage, route state, URL-safe temporary state, or an existing document store.
* Do not put large raw Markdown documents into public query parameters.
* Drag-and-drop must provide visible active and error states.
* File opening must work without uploading the file.
* Accept sensible Markdown file extensions already supported by the editor, likely `.md`, `.markdown`, and optionally `.txt`.
* Validate file type gently; do not reject a valid text Markdown file merely because a browser reports an empty or generic MIME type.
* Use accessible labels and keyboard-operable controls.
* Do not create a heavy editor toolbar in the homepage hero.
* On small screens, the input should remain easy to use and must not create horizontal overflow.

## Empty-state behaviour

The action button should be disabled or absent until meaningful content exists.

If “Try an example” is added, load one concise but realistic example and let the user continue through the same flow.

Do not silently overwrite an existing locally saved editor document without using the project’s established handling.
